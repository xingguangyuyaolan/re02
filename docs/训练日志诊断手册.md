# QMIX 训练日志诊断手册

适用范围：
- 训练入口 [train.py](train.py#L94)
- 算法日志来源 [src/scripts/attention_qmix.py](src/scripts/attention_qmix.py#L506)
- 环境日志来源 [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L48)
- 日志初始化 [src/scripts/logging_utils.py](src/scripts/logging_utils.py#L17)

本文目标：
- 看到日志就能快速判断是环境问题、训练问题还是配置问题
- 给出最短排查路径
- 给出修复后如何验收

---

## 1. 日志文件与优先级

每次运行会在运行目录下生成：
- 运行日志 [artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log)
- 错误日志 [artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/errors.log](artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/errors.log)
- 回合指标 [artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl](artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl)
- 汇总指标 [artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json](artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json)

推荐排查顺序：
1. 先看 summary.json 判断是否中断与中断类型
2. 再看 runtime.log 的第一段与最后一段
3. 再看 metrics.jsonl 判断训练是否真正开始和是否在学习
4. 最后看 errors.log 的堆栈细节

## 2. 60 秒快速分诊

按下面四问快速定位：

1. 是否一开始就中断
- 看 [artifacts/qmix/maze_qmix_seed0_20260410_145223/summary.json](artifacts/qmix/maze_qmix_seed0_20260410_145223/summary.json#L9)
- 若 interrupt_type 是 RuntimeError 且 reward_mean/loss_mean 为 null，优先判定环境 reset 或桥接问题。

2. 是否长期 loss 为 null
- 看 [artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl](artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl#L1)
- 前期 loss 为 null 可能正常热身；若持续很多回合仍 null，说明 replay buffer 未达到 batch 或回合异常短。

3. 是否开始训练但完全不收敛
- 看 episode 行与 [QMIX][Diag] 行，例如 [artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log#L42)
- 若 td_abs_mean、loss 长时间高位且 reward 无改善，优先调超参和奖励。

4. 是否只是人工停止
- KeyboardInterrupt 属于手动停机，不是故障，例如 [artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json](artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json#L9)

## 3. 症状到根因映射（最常用）

### 症状 A：启动即 RuntimeError，几乎没训练

典型信号：
- runtime.log 中出现 reset 失败
- 例如 [artifacts/qmix/maze_qmix_seed0_20260410_145223/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260410_145223/logs/runtime.log#L17)
- summary 里 interrupt_type 为 RuntimeError，loss_mean 为 null

高概率根因：
- Gazebo world reset 服务不可用或超时
- ROS2 与 Gazebo 桥接未正确起来
- world 名称与服务名不一致导致 reset 失败

代码定位：
- reset 失败抛错位置 [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L495)
- ROS reset 与 gz reset 回退逻辑 [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L416)

处理步骤：
1. 先确认 Gazebo world 正在运行且名称与训练配置一致
2. 确认桥接脚本在独立终端常驻运行 [src/scripts/setup_bridges.sh](src/scripts/setup_bridges.sh)
3. 优先用课程场景小步验证，比如 [configs/stage_1_open_train.json](configs/stage_1_open_train.json)
4. 如果偶发超时，先降低系统负载并重启仿真后再跑

修复验收：
- 能稳定通过至少 10 个 episode
- summary 中 interrupt_type 不再是 RuntimeError

---

### 症状 B：前几十回合 loss 为空

典型信号：
- metrics 前段连续出现 loss:null，例如 [artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl](artifacts/qmix/maze_qmix_seed0_20260409_223227/metrics.jsonl#L1)

解释：
- 这是回放缓存热身期，不一定是问题
- 训练器只有在 current_size >= batch_size 才更新，见 [src/scripts/attention_qmix.py](src/scripts/attention_qmix.py#L337)

何时判定异常：
- 回合数明显超过 batch_size 后仍几乎全是 loss:null
- 同时 benchmark_train_updates 长期为 0

处理步骤：
1. 检查 batch_size 是否过大（配置在 [configs/repro_nan_fix.json](configs/repro_nan_fix.json)）
2. 检查是否频繁中断导致还没达到训练门槛
3. 检查 episode 是否异常短（回合步数很小）

修复验收：
- metrics 出现稳定的非空 loss
- benchmark_train_updates 开始大于 0

---

### 症状 C：reward 长期为大负值，几乎不提升

典型信号：
- summary 中 reward_mean 很低，例如 [artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json](artifacts/qmix/maze_qmix_seed0_20260409_223227/summary.json#L13)
- 训练很久后 best_reward 仍接近负值

高概率根因：
- 难度与策略能力不匹配（直接在难场景开训）
- 探索衰减不合适，epsilon 长时间太高或下降太快
- 奖励比例不匹配（进度奖励被惩罚淹没）

处理步骤：
1. 必须走课程学习链：stage1 到 stage5
- [configs/stage_1_open_train.json](configs/stage_1_open_train.json)
- [configs/stage_2_easy_train.json](configs/stage_2_easy_train.json)
- [configs/stage_3_intermediate_train.json](configs/stage_3_intermediate_train.json)
- [configs/stage_4_challenging_train.json](configs/stage_4_challenging_train.json)
- [configs/stage_5_maze_like_train.json](configs/stage_5_maze_like_train.json)
2. 优先调三项：lr、epsilon_decay_steps、grad_clip_norm
3. 同步检查环境奖励参数来源 [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L552)

修复验收：
- 最近 50 回合平均 reward 比前 50 回合明显提升
- best_reward 持续刷新，不是只刷一次

---

### 症状 D：loss 波动很大或突然上冲

典型信号：
- runtime.log 出现大幅震荡 loss
- [QMIX][Diag] 中 td_abs_mean、q_eval_mean 与 q_target_mean 偏离明显
- 示例诊断行 [artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log#L42)

高概率根因：
- 学习率偏高
- 目标网络更新频率不合适
- 观测异常值传入导致目标值抖动

处理步骤：
1. 降低 lr（例如从 5e-4 到 3e-4 或 2.5e-4）
2. 复核 td_clip、q_clip、obs_clip（在配置中）
3. 观察 nan_skip_count 是否增长
- 指标字段写入位置 [src/scripts/attention_qmix.py](src/scripts/attention_qmix.py#L506)

修复验收：
- loss 的中位数下降，极端峰值减少
- td_abs_mean 从高位回落并趋稳

---

### 症状 E：警告 Potentially risky spawn layout

典型信号：
- runtime.log 出现 spawn 布局风险，例如 [artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260409_223227/logs/runtime.log#L9)

解释：
- 初始位置过近，可能导致训练初期大量碰撞与负奖励

处理步骤：
1. 调整 world 中 UAV 初始间距
2. 适当提高 startup_grace_steps，减弱起始抖动的碰撞影响
3. 检查最小安全高度参数是否合适

修复验收：
- 该警告消失或显著减少
- 前 20 回合极端负奖励次数下降

---

### 症状 F：日志显示 Reset service 名称不合法

典型信号：
- invalid ROS service name，例如 [artifacts/qmix/maze_qmix_seed0_20260410_145223/logs/runtime.log](artifacts/qmix/maze_qmix_seed0_20260410_145223/logs/runtime.log#L3)

解释：
- world 文件名含后缀，ROS 服务名语义与 Gazebo 服务名语义不完全一致
- 当前代码已做候选服务回退，但在某些组合下仍可能失败

代码定位：
- 服务名映射与候选列表构造 [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L48)

处理步骤：
1. 优先使用已验证可跑的 world 名称
2. 在同一终端重启仿真和桥接，避免僵尸服务
3. 若持续失败，先用 stage_1 场景回归测试

修复验收：
- reset 可连续成功多个回合
- 不再出现早期 RuntimeError

## 4. 指标阈值与判定建议

训练是否真正开始：
- benchmark_train_updates 在多数回合大于 0
- loss 不再长期为 null

环境是否成主要瓶颈：
- benchmark_env_step_time_sec 远高于 benchmark_train_update_time_sec
- 你的样本中环境时间通常显著更高，属于仿真瓶颈特征

是否出现数值风险：
- nan_skip_count 持续增长
- runtime 出现 Non-finite loss 或 Non-finite grad norm 警告
- 对应逻辑在 [src/scripts/attention_qmix.py](src/scripts/attention_qmix.py#L337)

是否是人为停止：
- summary interrupt_type 为 KeyboardInterrupt
- 这类中断不是训练故障

## 5. 标准排查流程（建议每次照做）

1. 读 summary
- 先看 interrupted 与 interrupt_type

2. 读 runtime 前 30 行
- 看配置、world、device、reset 警告

3. 读 runtime 末尾 30 行
- 看异常栈、是否保存 latest/final

4. 读 metrics 前 30 与后 30 行
- 看 loss 从 null 到非空的切换点
- 看 reward、epsilon、train_updates 变化

5. 如有数值疑点，定位训练更新函数
- [src/scripts/attention_qmix.py](src/scripts/attention_qmix.py#L337)

6. 如有 reset/桥接疑点，定位环境 reset 函数
- [src/scripts/gazebo_pettingzoo_env.py](src/scripts/gazebo_pettingzoo_env.py#L495)

## 6. 常见误判清单

误判 1：loss:null 就是代码坏了
- 纠正：常见是回放缓存尚未满足 batch_size。

误判 2：KeyboardInterrupt 是系统崩溃
- 纠正：多数是手动停止，模型和 summary 仍会在 finally 保存。

误判 3：reward 波动大就是训练失败
- 纠正：探索阶段波动大正常，要看移动平均和 best_reward 刷新趋势。

误判 4：只看 reward 不看 benchmark
- 纠正：很多时候瓶颈在仿真时间，不在神经网络。

## 7. 建议保留的最小运行记录

每次实验建议固定保留：
- 配置快照 config.json
- summary.json
- runtime.log
- metrics.jsonl
- best_model.pt 与 latest_model.pt

这些文件足够支持复盘、复现实验和参数回滚。
