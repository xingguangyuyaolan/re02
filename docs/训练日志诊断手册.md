# 训练日志诊断手册（当前版本）

更新时间：2026-04-20

## 1. 先看哪些文件

每次训练 run 目录下至少检查：

1. `summary.json`
2. `metrics.jsonl`
3. `logs/runtime.log`
4. `logs/errors.log`

建议顺序：先 `summary`，再 `runtime` 头尾，最后看 `metrics` 趋势。

## 2. 1 分钟快速分诊

1. `summary.json` 是否标记中断（exception/interrupted）？
2. `metrics.jsonl` 是否开始出现非空 `loss`？
3. `benchmark_train_updates` 是否长期为 0？
4. `collision_rate` / `out_of_bounds_rate` 是否长期高位？

## 3. 常见症状与处理

### 症状 A：训练刚开始就失败

高概率原因：世界未启动、桥接未常驻、reset 超时。

处理：

1. 确认 Gazebo 世界运行。
2. 确认 `src/scripts/setup_bridges.sh` 在独立终端常驻。
3. 先用 stage_1_open 配置回归测试。

### 症状 B：loss 长时间为空

可能原因：buffer 尚未达到 batch 门槛。

处理：

1. 检查 batch_size 与回合长度。
2. 检查是否频繁中断导致有效样本不足。

### 症状 C：reward 不提升且碰撞/出界高

可能原因：安全约束与奖励比例不匹配。

处理：

1. 优先看 `collision_rate`、`out_of_bounds_rate` 与 `coverage_rate` 的联动。
2. 检查边界惩罚、碰撞惩罚、生存奖励、epsilon 节奏。

### 症状 D：后期退化

可能原因：epsilon 过低后陷入局部坏策略。

处理：

1. 查看后 20% 回合指标与前 20% 对比。
2. 复核 `epsilon_decay_steps` 与阶段训练步数匹配。

## 4. 针对课程训练的排查

1. 每个阶段应写入独立 run 目录。
2. 跨阶段应“迁移权重、重置训练状态”。
3. Stage2/3 的初始 epsilon 应符合该阶段配置。

## 5. 验收口径

1. 训练可连续运行且不因 reset 异常中断。
2. `loss` 非空且更新计数稳定增长。
3. 覆盖指标有改善趋势，风险指标不持续恶化。
4. 评测脚本输出完整（summary + jsonl + 图）。
